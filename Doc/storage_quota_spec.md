# 存储配额与容量感知调度需求规格

## 背景与目标
ChunkForge的NodeInfo定义了capacity_bytes和used_bytes字段，但上传和修复流程从未检查这些值。节点无条件接受所有写入请求，直到磁盘物理满时写入失败。修复引擎选择目标节点时仅看分片数量，不考虑剩余容量，可能出现目标节点磁盘快满却仍被选中、导致修复写入失败的情况。

需要实现存储配额管理和容量感知调度：为节点设置配额上限，上传和修复在写入前检查目标节点容量，容量不足时拒绝写入或选择其他节点。

## 需求清单
- R-QUOTA-01: 支持为节点配置存储配额上限，作为软限制和硬限制
- R-QUOTA-02: 上传流程在分配分片前检查目标节点剩余容量，跳过容量不足的节点
- R-QUOTA-03: 修复引擎选择目标节点时综合考虑分片数和剩余容量
- R-QUOTA-04: 节点容量使用率超过软限制时发出警告事件
- R-QUOTA-05: 节点容量使用率超过硬限制时拒绝新分片写入
- R-QUOTA-06: 心跳上报中包含准确的磁盘使用量，后端据此更新元数据
- R-QUOTA-07: 集群整体容量不足时上传请求返回明确错误信息
- R-QUOTA-08: 前端节点卡片显示容量进度条和配额状态
- R-QUOTA-09: 管理员可动态调整节点配额，立即生效

## 现有架构约束
- 节点的心跳已上报disk_usage_bytes，但Coordinator只更新元数据未做容量判断
- 上传流程使用get_online_nodes()获取候选节点，需增加容量过滤
- 修复引擎使用get_node_with_least_shards()选择目标，需改为容量感知策略
- 节点的分片写入接口无条件接受，需增加容量预检

## 技术栈约束
- 后端：Python FastAPI + asyncio
- 节点存储：本地文件系统
- 心跳间隔：1秒
- 前端：React + D3.js

## 数据模型变更
- NodeInfo新增：quota_soft_limit_bytes (Optional[int]), quota_hard_limit_bytes (Optional[int])
- 新增 QuotaStatus 枚举：NORMAL / WARNING / CRITICAL
- NodeInfo扩展计算属性：quota_status (基于used_bytes/quota比例)
- 新增 EventType：NODE_QUOTA_WARNING / NODE_QUOTA_CRITICAL
- 上传/修复错误响应新增：INSUFFICIENT_CAPACITY 详情

## 流程设计
1. 节点注册时设置配额 → 心跳上报磁盘使用量 → 检查配额状态
2. 上传时：过滤掉达到硬限制的节点 → 在剩余节点中按分片数分配
3. 修复时：排除硬限制节点 → 在候选节点中按容量和分片数加权选择
4. 配额变更：管理员调整 → 立即更新元数据 → 下次调度生效

## 边界条件与异常处理
- 所有在线节点都达到硬限制：上传返回503，修复标记DEGRADED
- 节点上报的磁盘用量与实际不一致：以节点上报值为准，但写入时节点再次校验
- 配额调整到低于当前使用量：不驱逐已有分片，但阻止新写入
- 修复期间目标节点达到硬限制：选择下一个可用节点

## 验收标准
- 节点配额达到硬限制时新上传的分片不写入该节点
- 修复引擎优先选择容量充足的节点
- 软限制触发时前端显示警告
- 集群整体容量不足时上传返回明确错误
- 配额调整后立即影响下一次写入决策
