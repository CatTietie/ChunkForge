# 文件覆盖写入与版本管理需求规格

## 背景与目标
ChunkForge当前每次上传都生成新的file_id，同名文件会创建独立记录。旧文件的分片永远不会被清理，节点磁盘空间只增不减。用户更新文件后，文件列表里出现同名条目，下载时可能拿到旧版本。

需要支持文件覆盖写入：上传同名文件时替换旧分片而非创建新记录，同时可选保留历史版本供回溯。

## 需求清单
- R-VER-01: 上传时如果同名文件已存在，默认覆盖旧文件（替换所有分片）
- R-VER-02: 覆盖写入必须原子性：新分片全部写入成功后才删除旧分片，任何一步失败则回滚
- R-VER-03: 可配置版本保留策略：保留最近N个版本或按时间保留
- R-VER-04: 历史版本可通过API查询和下载
- R-VER-05: 文件元数据中记录版本号和版本历史
- R-VER-06: 旧版本的孤立分片在版本过期后自动清理
- R-VER-07: 修复引擎需要感知多版本：同一分片索引存在多个版本时，修复当前版本
- R-VER-08: 通过WebSocket推送版本变更事件
- R-VER-09: 前端文件管理支持版本列表查看和历史版本下载

## 现有架构约束
- 上传流程中分片写入和元数据注册是顺序执行的，覆盖时需要在旧分片删除前完成新分片写入
- 下载流程根据file_id查询元数据，版本化需要区分当前版本和历史版本
- 修复引擎通过file_id定位分片，版本化后需要确保修复操作针对正确的版本
- 节点存储按file_id分目录，同一文件的不同版本需要不同的file_id或子目录

## 技术栈约束
- 后端：Python FastAPI + asyncio
- RS(4+2)纠删码：固定分片数
- 节点存储：本地文件系统，按file_id/shard_id.shard组织
- 前端：React

## 数据模型变更
- FileMeta新增：version (int), is_current (bool), parent_file_id (Optional[str])
- 新增 FileVersion 模型：version, file_id, upload_time, original_checksum, size
- 新增 EventType：FILE_VERSION_CREATED / FILE_VERSION_EXPIRED
- ShardLocation可引用不同file_id的分片（版本化后同一逻辑文件对应多个物理file_id）

## 流程设计
1. 上传同名文件 → 编码新分片 → 写入节点 → 验证 → 注册新版本元数据 → 标记旧版本为历史
2. 原子性保证：先写新分片，全部成功后更新元数据，最后异步删除旧分片
3. 版本清理：后台任务定期扫描过期版本，删除对应分片和元数据

## 边界条件与异常处理
- 覆盖写入期间旧版本分片所在节点宕机：新版本写入不受影响，旧版本分片异步清理
- 新分片部分写入失败：回滚已写入的新分片，保留旧版本不变
- 版本清理期间用户下载历史版本：下载完成后再删除分片
- 修复与覆盖并发：修复操作必须识别当前版本的file_id

## 验收标准
- 上传同名文件后下载得到最新内容
- 覆盖过程中下载仍返回旧版本，覆盖完成后返回新版本
- 部分新分片写入失败时旧版本完好无损
- 历史版本可查看和下载
- 过期版本的孤立分片被自动清理
