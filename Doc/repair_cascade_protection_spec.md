# 修复优先级与级联故障防护需求规格

## 背景与目标
ChunkForge当前的修复引擎在多节点同时宕机时，对所有受影响文件并发执行修复，没有优先级排序。修复过程中如果又出现节点宕机，新修复的分片可能恰好位于刚宕机的节点上，导致修复白做。修复引擎不检测也不防护级联故障场景。

需要实现修复优先级调度和级联故障防护：根据文件重要性和风险等级排序修复顺序，检测级联故障并暂停修复，避免在集群不稳定期间浪费修复资源。

## 需求清单
- R-CASCADE-01: 修复任务入队时计算优先级：文件分片数越接近阈值越优先，DEGRADED状态文件优先于HEALTHY
- R-CASCADE-02: 修复引擎从并发执行改为优先级队列调度，按优先级逐批执行
- R-CASCADE-03: 检测级联故障：修复期间新节点宕机时，暂停当前所有修复任务
- R-CASCADE-04: 级联暂停后等待集群稳定（无新节点宕机超过N秒），再按新优先级恢复修复
- R-CASCADE-05: 修复前预检查目标节点可用性，避免将修复分片写入即将宕机的节点
- R-CASCADE-06: 连续级联故障超过阈值时标记集群为UNSTABLE状态，拒绝新上传
- R-CASCADE-07: 修复过程中实时更新剩余风险：哪些文件还差几个分片就丢失
- R-CASCADE-08: 通过WebSocket推送修复优先级变化和级联告警事件
- R-CASCADE-09: 前端显示修复队列、优先级排序和级联状态指示器
- R-CASCADE-10: 修复完成后检查是否有文件因级联故障变为LOST，主动推送告警

## 现有架构约束
- 修复引擎使用asyncio.gather并发所有文件修复，需改为优先级队列
- 心跳监控检测到节点宕机后直接调用on_node_dead，需增加级联判断
- Term机制在每次宕机时递增，可用于判断集群不稳定程度
- 修复引擎的信号量限流机制可保留，但需配合优先级调度

## 技术栈约束
- 后端：Python FastAPI + asyncio
- 并发模型：asyncio + Semaphore
- 心跳超时：3秒
- 前端：React

## 数据模型变更
- 新增 ClusterHealth 枚举：STABLE / DEGRADED / UNSTABLE
- 新增 RepairPriority 模型：file_id, priority_score, remaining_survivors, risk_level
- RepairTask新增：priority (float), batch_id (str)
- 新增 EventType：REPAIR_PAUSED / REPAIR_RESUMED / CASCADE_DETECTED / CLUSTER_UNSTABLE / RISK_UPDATED
- MetadataStore扩展：cluster_health (ClusterHealth)

## 流程设计
1. 节点宕机 → 收集受影响文件 → 计算各文件优先级 → 入优先级队列
2. 检查集群健康度 → UNSTABLE则暂停修复 → 否则按优先级逐批执行
3. 修复期间新节点宕机 → 检测级联 → 暂停当前修复 → 重新计算优先级 → 等待稳定
4. 稳定后恢复修复 → 执行完一批后检查集群状态 → 决定是否继续

## 边界条件与异常处理
- 修复暂停期间文件被删除：从队列移除
- 等待集群稳定期间又出现宕机：重置稳定计时器
- 所有在线节点都在修复队列中作为目标节点：降级为串行修复
- 优先级计算中多个文件同等优先级：按文件上传时间排序（老文件优先）

## 验收标准
- 多节点同时宕机时，分片最接近丢失阈值的文件最先修复
- 修复期间新节点宕机时，修复自动暂停并推送级联告警
- 集群稳定后修复自动恢复，优先级重新计算
- 连续级联故障时集群标记为UNSTABLE，新上传被拒绝
- 前端显示修复队列和优先级排序
