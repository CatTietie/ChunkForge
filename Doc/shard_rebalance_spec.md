# 分片再平衡引擎需求规格

## 背景与目标
ChunkForge当前分片放置仅在文件上传时执行（最少分片优先），修复时也采用相同策略。新节点加入集群后，已有分片不会自动迁移，导致热力图长期不均衡。随着文件不断上传和节点增减，部分节点承载的分片数远超其他节点，写入和读取压力集中在少数节点上。

需要实现后台再平衡引擎，在集群拓扑变化或分片分布不均时，自动触发增量分片迁移，使各节点分片数趋于均衡。

## 需求清单
- R-REBAL-01: 定义再平衡触发条件：新节点加入、节点恢复在线、手动触发、定时巡检
- R-REBAL-02: 再平衡引擎作为后台异步任务运行，不阻塞上传和下载
- R-REBAL-03: 迁移采用增量策略：每轮选择迁移收益最大的N个分片（N可配置）
- R-REBAL-04: 分片迁移流程：从源节点读取 → 写入目标节点 → 验证校验和 → 更新元数据 → 删除源节点分片
- R-REBAL-05: 迁移过程中节点宕机的处理：源节点宕机则跳过该分片，目标节点宕机则重选目标
- R-REBAL-06: 集群状态不稳定时暂停再平衡（如修复进行中、节点数少于k+m）
- R-REBAL-07: 通过WebSocket推送再平衡进度事件
- R-REBAL-08: 前端热力图实时反映分片迁移进度

## 现有架构约束
- 统一后端：Gateway + Coordinator 共享 MetadataStore
- 分片传输协议：PUT/GET/DELETE /shards/{id}，带校验和验证
- 修复引擎已有分片重建和写入逻辑，可复用部分机制
- 事件总线通过WebSocket广播到前端

## 技术栈约束
- 后端：Python FastAPI + asyncio
- 前端：React + D3.js
- 节点间通信：httpx AsyncClient
- 状态持久化：JSON文件

## 数据模型变更
- 新增 RebalanceTask 模型：task_id, source_node_id, target_node_id, shard_ids, state, progress
- 新增 RebalanceState 枚举：PENDING / MIGRATING / VERIFYING / COMPLETED / PAUSED / CANCELLED
- 新增 EventType：REBALANCE_STARTED / REBALANCE_PROGRESS / REBALANCE_COMPLETED / REBALANCE_PAUSED

## 引擎流程设计
1. 触发检查 → 计算各节点分片偏差 → 偏差超过阈值时启动再平衡
2. 选择迁移对（源节点→目标节点，分片列表）→ 入队
3. 逐分片迁移：读取 → 写入 → 验证 → 更新元数据 → 删除源分片
4. 全部完成后发布完成事件
5. 任何时刻检测到集群不稳定则暂停

## 边界条件与异常处理
- 迁移期间文件被删除：跳过该分片，继续其余分片
- 迁移期间源分片被修复引擎移动：检测到校验和不匹配则跳过
- 目标节点磁盘满：选择下一个目标节点
- 再平衡与修复并发：修复优先，再平衡自动暂停

## 验收标准
- 新节点加入后，再平衡引擎在合理时间内将分片迁移到新节点
- 迁移过程中上传下载不受影响
- 迁移过程中源节点宕机，已迁移分片不丢失
- 修复进行中时再平衡自动暂停
- 前端热力图实时显示分片位置变化
